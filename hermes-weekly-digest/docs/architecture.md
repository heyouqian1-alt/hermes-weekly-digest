# 系统架构

## 为什么要做这个项目

Hermes Agent 本身是一个强大的多 profile AI 框架，但存在一个盲区：

> **你有多个 agent，每个有自己的会话记录，但没有一个地方能统览它们这周都做了什么。**

Hermes 提供了 `/journey` 查看单个 agent 的学习轨迹，但跨 profile 的语义化总结只能靠手动拼接。这个项目填补了这个空白。

## 关键设计决策

### 1. 直接读 SQLite，不用 API

Hermes 的会话存储在 `state.db`（SQLite 数据库），文件格式稳定。直接读 DB 比用命令行 `hermes sessions list` 快 10 倍以上，且能获取更多细节（如用户消息原文）。

### 2. 两步走：收集 + 生成

- **收集阶段**（Python 脚本）：无 token 消耗，纯 SQL + 文件扫描
- **生成阶段**（LLM）：只有周一才触发一次，耗费约 5K-10K tokens

如果合并成一步，每周一都要消耗大量 tokens 扫描数据。两步走让 cron 可以高频（每 4 小时）运行而几乎零开销。

### 3. 周一守护，而非定时定日

```python
if now.weekday() != 0:  # 不是周一
    return  # 静默退出
```

相比 "每周一 10:00" 的固定时间，周一守护 + 每 4 小时触发可以覆盖任何开机时间——周一早上 9 点开机，13 点就能收到报告；下午 3 点开机，19 点也能收到。不会因为关机而错过。

### 4. 亚麻纸 PDF

`md-to-pdf.py` 用 Pillow 实时生成纺织纹理：

1. 200x200 像素 tile：经线（水平）+ 纬线（垂直）+ 随机噪点
2. GaussianBlur(0.6) 柔化 → 织物质感
3. 平铺到 A4 全页
4. 叠加文字

纹理只生成一次，缓存为 PNG，后续零开销。

## 依赖关系

```
weekly-digest.py
  ├── Python 标准库（sqlite3, json, datetime, os, pathlib）
  └── PyYAML（可选，用于 config.yaml）

md-to-pdf.py
  ├── fpdf2（PDF 生成）
  ├── Pillow（纹理生成）
  └── Windows 系统字体（微软雅黑）
```

## 扩展性

### 增加新的数据源

在 `collect_profile_data()` 中添加新的扫描逻辑即可。例如扫描 `plugins/` 目录、读取 `gateway_state.json`、分析 token 用量等。

### 增加新的输出格式

新增一个转换脚本，比如 `md-to-html.py`、`md-to-docx.py`，然后在 workflow 中调用。

### 适配其他 AI Agent 框架

只要目标框架的会话数据存储在 SQLite 或可读文件中，修改 `collect_profile_data()` 即可适配。
